OpenShift AI and llm-d: The Enterprise Path to Distributed LLM Inference

Markell Rawls June 2026 ~12 min read

Introduction

There is a familiar pattern in enterprise technology: a demo that runs beautifully on a single machine crumbles the moment it faces real production traffic. Large language model inference is no exception. Getting a model to respond on your laptop is straightforward. Getting that same model to serve 10,000 concurrent users with sub-second time-to-first-token, guaranteed uptime, automatic scaling, and GPU cost efficiency is an entirely different engineering challenge.

The gap between "it works" and "it works at scale with SLAs" is where most organizations stall. Building production inference infrastructure from scratch means solving a cascade of hard problems: intelligent request routing, KV-cache management across distributed GPU pools, disaggregated prefill and decode scheduling, GPU topology-aware placement, autoscaling that responds to inference-specific signals rather than generic CPU metrics, and monitoring that surfaces the metrics that actually matter for LLM workloads.

This is precisely the space where the upstream-to-enterprise pipeline proves its value. llm-d is the CNCF Sandbox open-source project purpose-built for distributed LLM inference routing. Red Hat AI (formerly OpenShift AI) is the enterprise product that takes llm-d upstream and makes it production-ready with support contracts, hardware certification, and deep integration across the Red Hat ecosystem. Together, they represent the most complete path from open-source innovation to enterprise-grade LLM infrastructure available today.

What Is Red Hat AI?

Red Hat AI is Red Hat's enterprise AI platform built on top of OpenShift, Red Hat's Kubernetes distribution. Previously branded as "OpenShift AI," it was rebranded to "Red Hat AI" to reflect a scope that extends well beyond the container orchestration layer into a full-lifecycle AI platform.

At its core, Red Hat AI brings together several capabilities under one unified platform:

  • Model Serving — Powered by vLLM and llm-d under the hood, providing high-performance inference with intelligent routing, disaggregated serving, and KV-cache-aware scheduling. This component is officially called the Red Hat AI Inference Server.
  • Model Registry — A centralized catalog for managing model artifacts, versions, and metadata across an organization.
  • Training via InstructLab — An open-source approach to model fine-tuning that allows domain experts to contribute knowledge without needing deep ML expertise.
  • GPU Management — Automatic GPU driver management, topology detection, and workload placement through the NVIDIA GPU Operator and Node Feature Discovery (NFD).

Everything in Red Hat AI is Kubernetes-native. Models are deployed through Custom Resource Definitions (CRDs). Lifecycle management happens through operators and controllers. Monitoring flows through the same Prometheus and Grafana stack that teams already use for their other OpenShift workloads. This is not an AI platform bolted onto Kubernetes; it is an AI platform built as Kubernetes.

The Upstream-to-Enterprise Pipeline

If you have worked with Red Hat technologies before, the relationship between llm-d and Red Hat AI will feel familiar. It follows the same model that has powered enterprise open source for decades:

Linux kernel becomes RHEL. Kubernetes becomes OpenShift. llm-d becomes Red Hat AI Inference Server.

llm-d is the upstream CNCF Sandbox project. It is community-driven, fast-moving, and where bleeding-edge features land first. New routing strategies, experimental optimizations, and novel approaches to KV-cache management all debut in the llm-d project. The community iterates quickly, and the API surface evolves as the project learns from real-world deployments.

Red Hat AI takes llm-d at specific points in its release cycle, certifies it against specific hardware configurations, tests it through Red Hat's extensive QE process, adds enterprise features like FIPS compliance and air-gapped deployment support, and ships it with 24/7 support. When customers deploy the Red Hat AI Inference Server, they are running llm-d with the confidence that it has been tested on their specific GPU hardware, validated against their OpenShift version, and backed by a support organization that can respond at 3 AM on a Saturday.

The pipeline flows both directions. Red Hat engineers are active contributors to the llm-d upstream project. Bug fixes discovered through enterprise testing flow back to the community. Performance optimizations validated at scale in customer environments become upstream features. The LLMInferenceService CRD itself was designed collaboratively to be the Kubernetes-native standard for deploying llm-d workloads, whether on vanilla Kubernetes or on OpenShift.

This bidirectional flow is what separates the Red Hat model from a simple vendor fork. The upstream project gets stronger because of enterprise feedback, and the enterprise product stays current because it tracks a healthy, active upstream community.

Key Integration Points

KServe and the LLMInferenceService CRD

KServe is the standard model serving framework on Kubernetes, providing a consistent API for deploying and managing machine learning models. Red Hat AI uses KServe under the hood as the foundation for its model serving capabilities.

The primary API for deploying inference workloads is the LLMInferenceService CRD. This custom resource encapsulates everything needed to run a distributed LLM inference deployment: the model source, serving runtime configuration, GPU requirements, disaggregated serving topology, routing strategy, and autoscaling parameters. A single YAML file replaces what would otherwise be dozens of manually coordinated resources.

yaml llminferenceservice.yaml 
apiVersion: serving.llm-d.ai/v1alpha1
kind: LLMInferenceService
metadata:
  name: llama-3-70b
  namespace: ai-inference
spec:
  model:
    name: meta-llama/Llama-3-70B-Instruct
    source:
      uri: pvc://model-store/llama-3-70b
  serving:
    runtime: vllm
    replicas: 4
    resources:
      gpu: nvidia.com/gpu
      gpuCount: 4
      gpuType: H100-SXM
    disaggregated:
      enabled: true
      prefillReplicas: 2
      decodeReplicas: 2
  routing:
    endpoint: EPP
    strategy: kv-cache-aware
  autoscaling:
    minReplicas: 2
    maxReplicas: 8
    targetUtilization: 70

This YAML tells the system everything it needs to know: pull the Llama 3 70B model from a persistent volume, run it on vLLM across four replicas with H100 GPUs, split those replicas into a disaggregated prefill/decode topology, route requests through the Endpoint Picker (EPP) with KV-cache-aware scheduling, and autoscale between two and eight replicas based on utilization. The operator handles the rest — creating pods, services, configuring networking, and wiring up the routing layer.

GPU Operator and Node Feature Discovery

Distributed LLM inference is fundamentally a GPU orchestration problem, and getting GPU placement right is non-negotiable. The OpenShift GPU Operator manages NVIDIA GPU drivers and the container runtime hooks needed for GPU passthrough into containers. It ensures the correct driver version is installed, CUDA libraries are available, and device plugins are registered with the kubelet.

Node Feature Discovery (NFD) works alongside the GPU Operator to detect and label the GPU topology of every node in the cluster. It identifies GPU models, memory capacity, NVLink connectivity between GPUs, PCIe topology, and NUMA node affinity. These labels become the building blocks for intelligent scheduling decisions.

Together, GPU Operator and NFD ensure that when an LLMInferenceService requests four H100 GPUs with tensor parallelism, the scheduler places the pod on a node where those four GPUs are connected via NVLink for maximum inter-GPU bandwidth. This is critical for disaggregated serving, where prefill and decode workloads have very different GPU communication patterns and placing them on the wrong topology can destroy performance.

Monitoring Integration

LLM inference generates a unique set of operational signals that generic application monitoring simply does not capture. OpenShift's built-in Prometheus instance scrapes llm-d metrics natively, providing visibility into the metrics that matter:

  • Time-to-first-token (TTFT) — The latency a user experiences before the first token appears. This is the primary responsiveness metric.
  • Inter-token latency (ITL) — The time between successive tokens during generation. Spikes here cause visible "stuttering" in streaming responses.
  • Queue depth — The number of requests waiting for a free inference slot. This is the leading indicator for when scaling is needed.
  • KV-cache utilization — How much of the available KV-cache memory is consumed. When this saturates, new requests must wait or be routed elsewhere.
  • GPU utilization and memory — Standard GPU metrics that, when correlated with inference metrics, reveal whether the system is compute-bound or memory-bound.

Grafana dashboards purpose-built for inference observability ship with Red Hat AI, providing a single pane of glass for the entire inference stack. Alert rules can be configured for SLA violations — for example, triggering a page when p99 TTFT exceeds 500ms or when KV-cache utilization on any replica exceeds 90%.

Walking Through a Deployment

Let's trace through what it actually looks like to go from an empty OpenShift cluster to a production inference endpoint. These are the concrete steps an operator takes:

Install the Red Hat AI Operator from OperatorHub. This deploys the controllers that understand the LLMInferenceService CRD and manage the inference lifecycle.
Configure GPU Operator and NFD. The GPU Operator installs NVIDIA drivers on GPU nodes. NFD labels those nodes with GPU topology information. Both are available as OpenShift operators.
Create a model storage PVC and download the model. The model weights are stored on a PersistentVolumeClaim backed by high-throughput storage (ideally NVMe-backed). Model download can be handled by a pre-pull job or by referencing a container registry.
Apply the LLMInferenceService CRD. Submit the YAML resource shown earlier. The operator validates the configuration against available cluster resources.
The operator creates the full stack: vLLM inference pods (split into prefill and decode if disaggregated), the EPP routing deployment, a Kubernetes Service for internal traffic, and an OpenShift Route for external access.
Test with curl to the OpenShift Route. The endpoint exposes an OpenAI-compatible API, so any existing client library or tool that speaks the OpenAI chat completions protocol works out of the box.
Monitor via OpenShift Console dashboards. The built-in dashboards show inference metrics alongside cluster health, providing operational visibility from day one.

What is notable about this workflow is what the operator does not have to do: there is no manual GPU driver installation, no hand-written Kubernetes networking configuration, no custom Prometheus scrape configs, and no bespoke load balancer setup. The operator and CRD abstract away the complexity that would otherwise consume weeks of engineering time.

Enterprise vs. Upstream: Different Tools for Different Stages

A common misconception is that choosing between upstream and enterprise is a binary decision. In practice, the most effective organizations use both, choosing the right tool for each stage of their workflow.

Upstream llm-d

  • Latest features, often weeks ahead of enterprise
  • Community-driven support via GitHub and Slack
  • Bring your own Kubernetes cluster
  • Manual GPU driver and topology setup
  • DIY monitoring and alerting
  • Ideal for development and experimentation
  • Rapid iteration on routing strategies
  • Full flexibility to customize and extend

Red Hat AI

  • Certified hardware profiles tested by Red Hat QE
  • 24/7 enterprise support with SLA guarantees
  • Automatic GPU driver management
  • Integrated monitoring, logging, and alerting
  • Security patches and CVE response
  • FIPS compliance for regulated industries
  • Air-gapped deployment for disconnected environments
  • Certified and signed container images

The sweet spot for many organizations is to use upstream llm-d in development and staging environments where engineers need access to the latest features and the freedom to experiment with different configurations. When a workload is ready for production — when it needs to serve real users with contractual SLAs — it moves to Red Hat AI, where the same CRD-based API ensures that the deployment manifest transfers directly from staging to production with minimal changes.

This is not about upstream being inferior. It is about recognizing that production infrastructure has requirements that extend far beyond feature functionality. When an inference endpoint is serving a customer-facing application at 3 AM and something breaks, the difference between "file a GitHub issue and wait" and "call Red Hat support and get an engineer on the phone" can mean the difference between a minor incident and a major outage.

Why This Architecture Matters Now

LLM inference is transitioning from an experimental capability to critical infrastructure. Organizations are embedding LLM-powered features into their core products — customer support agents, code generation tools, document analysis pipelines, and real-time decision systems. When these features go down, revenue stops.

The infrastructure patterns that served the early days of LLM adoption — single-GPU deployments behind a basic load balancer — cannot meet these demands. The request patterns are unique: long-running streaming connections, variable-length outputs, memory-intensive KV-cache state, and bursty traffic patterns that make traditional autoscaling heuristics unreliable.

llm-d was built specifically for these challenges. Its EPP (Endpoint Picker) routing layer understands inference semantics: it knows which replicas have warm KV-cache entries, which are running prefill versus decode, which have available GPU memory, and which are about to complete long-running requests. This is not generic HTTP load balancing — it is inference-aware traffic management.

Red Hat AI takes these capabilities and wraps them in the operational framework that enterprise teams need: a single operator that manages the full stack, monitoring that speaks the language of inference metrics, GPU management that handles driver lifecycle without manual intervention, and a support organization that understands the entire stack from GPU silicon to API response.

For organizations evaluating their inference infrastructure strategy, the llm-d-to-Red-Hat-AI pipeline offers a compelling path. Start with upstream for speed and experimentation. Graduate to enterprise for stability and support. Contribute improvements back to the community. This is the same model that made Linux the foundation of the internet and Kubernetes the foundation of cloud computing. It is now making llm-d the foundation of enterprise LLM inference.

Conclusion

The relationship between llm-d and Red Hat AI represents the best of open source operating at its full potential. Innovation happens in the community, where researchers, engineers, and operators collaborate on the hardest problems in distributed inference. Enterprise hardening happens at Red Hat, where that innovation is tested, certified, and backed by the support that production workloads demand.

As LLM inference becomes critical infrastructure — as fundamental to the enterprise as databases and message queues — this model ensures that organizations can move fast without sacrificing reliability. They can adopt cutting-edge routing strategies without taking on the operational risk of running unsupported software. They can run on the latest GPU hardware knowing that someone has tested the full stack before it reaches their cluster.

The enterprise path to distributed LLM inference is not about choosing between open source and commercial software. It is about using both, at the right stage, with the right expectations. llm-d and Red Hat AI make that path concrete.
